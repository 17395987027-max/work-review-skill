# 工作复盘

## 首次安装自检（每次被调用时执行）

在开始复盘之前，先检查用户的 `~/.claude/CLAUDE.md` 是否包含"自动复盘规则"段落。

**如果不存在**：向用户说明：
> 这个 Skill 需要将触发规则写入 CLAUDE.md 才能自动运行。写入内容仅 5 行，不影响其他功能。是否允许？

用户同意后，在 CLAUDE.md 的"常用操作快速参考"段落后插入以下内容：

```
## 自动复盘规则

遇到以下情况，任务结束后主动复盘：
1. 同一任务 ≥ 2 次仍不满足，或同一问题试了 ≥ 3 种方案
2. 用户说了"太慢/又错/怎么还没好/上次不是"等不满信号

复盘流程：定位问题来源（哪个 Skill / 通用问题）→ 提炼教训 → 询问用户 → 确认后写回对应位置（Skill 的 SKILL.md 或 CLAUDE.md）→ 告知用户。
```

写入后告知用户："已就绪，下次遇到重复踩坑会自动复盘。"

**如果已存在**：跳过自检，直接进入复盘流程。

---

## 触发条件

以下任一条件满足时激活：

1. 同一任务反复执行 ≥ 2 次，用户仍然不满意
2. 同一问题尝试了 ≥ 3 种不同方案才解决
3. 用户明确说"太慢了""怎么还没好""又错了"等不满信号

## 执行流程

### 1. 观察阶段（任务进行中）
静默记录：
- 任务目标是什么
- 试了哪些方案、为什么失败
- 最终怎么解决的（或放弃的）

### 2. 复盘阶段（任务结束后）
任务解决或用户放弃后，执行：

```bash
# 查看当前记忆文档行数
wc -l ~/.claude/CLAUDE.md
```

然后决定写入方式：

### 3. 定位问题来源

复盘时判断问题属于哪种，写入不同位置：

- **某个 Skill 不好用**（如 jessup-research 的搜索策略错了）→ 写回该 Skill 的 SKILL.md，优化它的规则
- **跨 Skill 的通用问题**（如网络策略、工具选择）→ 写入 CLAUDE.md 核心陷阱表

判断方法：回顾失败过程，如果用户明确调用了某个 Skill 做了某件事但效果不好，问题归属该 Skill。

### 4. 写入规则（严格限制）

| 教训类型 | 写入位置 | 格式 | 上限 |
|----------|----------|------|------|
| **Skill 自身缺陷** | 该 Skill 的 SKILL.md | 优化触发条件/执行流程/写入策略 | 3 行 |
| **新 Bug 级陷阱** | CLAUDE.md 核心陷阱表 | 新增一行 | 1 行 |
| **已有陷阱的变体** | 更新原行解决方案 | 合并到已有行 | 不新增 |
| **新工具/命令** | CLAUDE.md 常用操作快速参考 | 新增一行 | 1 行 |
| **搜索/网络相关** | CLAUDE.md 搜索策略 | 一句话 | 1 行 |
| **重复内容** | 不写入 | 已有记录则跳过 | — |

**核心原则**：
- 每次复盘最多新增 **3 行**（1 行给 CLAUDE.md + 2 行给 Skill）
- 优先写入出问题的 Skill 自身，其次写 CLAUDE.md
- 如果 CLAUDE.md 已超 80 行，写入前先删旧条目
- 格式必须简洁，禁止代码块、长篇解释、时间戳

### 5. 用户确认
写入前展示修改计划：
```
## 复盘建议
- [优化 jessup-research] 搜索策略增加百度移动版兜底
- [新增陷阱] **Cloudflare 403** | 优先 treaties.un.org
- [无需新增] epub 转换已有记录

写入以上修改？(y/n)
```
用户确认后再执行写入。

### 6. 写入后告知
写入完成后，向用户简短告知改动内容，例如：
> 已写入记忆：**epub→pdf** → 直接 calibre，别绕路。

一句话说清"改了哪里"，不重复展示整段代码。用户完全不知情的情况下不写入任何内容。

## 跨 Skill 优化

复盘不限于改 CLAUDE.md。如果问题出在某个 Skill 上（用户调用了它但效果不好），直接把优化写进那个 Skill 的 SKILL.md。典型场景：

- `jessup-research` 搜索策略只写了 Bing，国内用不了 → 补上 360/百度移动
- `pdf-converter` 没写 calibre 兜底，反复试 pandoc/weasyprint → 加首选方案
- 某个 Skill 的触发条件太宽或太窄 → 调整阈值

也包括改写自己（work-review SKILL.md）：
- 触发频率太高 → 收紧
- 该记的漏了 → 降低门槛
- 写入策略有漏洞 → 优化规则

改写任何 Skill 前必须征得用户同意，写入后告知。

## 不复盘的情况
- 任务一次性顺利完成
- 失败原因是外部不可控（网络断开、API 限流）
- 纯用户原因（文件不存在、需求变更）
- 教训已在 CLAUDE.md 中有记录
